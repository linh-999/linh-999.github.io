Executive Summary

Project này biến một email nghi phishing thành một case điều tra có cấu trúc. Nội dung tập trung vào các câu hỏi SOC thực tế: người gửi có đáng tin không, domain có mới/giả mạo không, URL dẫn tới đâu, attachment có hash đáng ngờ không, user đã click/chạy file chưa và cần quarantine hay escalate.

Objective & Scope

  • Chuẩn hóa quy trình tiếp nhận email nghi phishing từ người dùng.
  • Tách IOC gồm sender, reply-to, return-path, URL, domain, IP, attachment hash.
  • Đưa ra verdict dựa trên evidence thay vì cảm tính.
  • Đề xuất action: quarantine, block sender/domain, reset password, user awareness hoặc close as benign.

Architecture / Log Flow

Alert / Event Source→Log Collection→SIEM / Dashboard→Analyst Triage→Verdict & Response
  • User báo cáo email nghi ngờ.
  • SOC lấy header, body, link và attachment metadata.
  • IOC được kiểm tra qua reputation/sandbox/threat intel.
  • Kết quả được ghi vào phishing triage form.
  • Action được đề xuất tùy mức độ rủi ro và user interaction.
Evidence artifacts: topology, dashboard, rule detail, raw event và command output được quản lý như bằng chứng kỹ thuật của case.

Detection Logic / Use Cases

  • Sender spoofing: From khác Return-Path/Reply-To bất thường.
  • Credential harvesting: link dẫn tới trang login giả hoặc domain gần giống thương hiệu.
  • Malicious attachment: file nén, macro, executable hoặc hash bị đánh dấu.
  • Business Email Compromise: nội dung thúc ép chuyển tiền, đổi tài khoản hoặc gửi thông tin nhạy cảm.
# analyst-note
Detection should define: objective → data source → condition → expected evidence → false-positive notes → response action.

Investigation Workflow

  1. Đọc subject, sender và người nhận để xác định bối cảnh.
  2. Phân tích header: SPF/DKIM/DMARC nếu có, received chain, reply-to.
  3. Extract URL an toàn, kiểm tra domain, redirect và reputation.
  4. Kiểm tra attachment hash/extension, không mở trực tiếp trên máy thật.
  5. Xác định user interaction: đã click, nhập mật khẩu, tải/chạy file hay chưa.
  6. Ghi verdict và action theo confidence.

Key Findings

  • Phishing triage cần kết hợp technical IOC và ngữ cảnh nội dung.
  • Không nên chỉ dựa vào giao diện email vì tên hiển thị có thể giả mạo.
  • User interaction là yếu tố quyết định escalation.

Recommendations

  • Chặn domain/sender nếu có evidence rõ.
  • Reset password và revoke session nếu user nhập credential.
  • Gửi cảnh báo awareness nếu nhiều user nhận cùng chiến dịch.
  • Cập nhật mail gateway rule và IOC watchlist.

Limitations & Next Improvements

  • Tạo form triage cố định để ghi evidence nhanh.
  • Tự động extract URL/hash từ email.
  • Tích hợp sandbox cho attachment trong môi trường cô lập.

Limitation được ghi rõ để tránh phóng đại kết quả lab. Khi có ảnh/log thật, phần này nên được cập nhật bằng evidence cụ thể.
