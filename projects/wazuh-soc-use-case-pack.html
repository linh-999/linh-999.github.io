Executive Summary

Project mô phỏng cách một SOC xây dựng use case trên Wazuh: xác định mục tiêu phát hiện, nguồn log cần thu thập, điều kiện rule, ví dụ alert, cách analyst điều tra và điểm cần tuning. Nội dung được viết theo hướng “detection content” chứ không chỉ hướng dẫn cài Wazuh.

Objective & Scope

  • Xây dựng use case giám sát thay đổi file nhạy cảm.
  • Theo dõi failed login, sudo command và suspicious command line.
  • Mapping alert sang MITRE ATT&CK để tăng giá trị phân tích.
  • Tạo template report cho từng alert: context, evidence, verdict, recommended response.

Architecture / Log Flow

Alert / Event Source→Log Collection→SIEM / Dashboard→Analyst Triage→Verdict & Response
  • Wazuh agent cài trên Linux endpoint.
  • Agent thu thập auth log, FIM event và command/audit log.
  • Wazuh manager decode, rule match và gán severity.
  • Indexer lưu dữ liệu phục vụ search và dashboard.
  • Dashboard hiển thị alert để analyst triage.
Chèn ảnh lab tại đây: topology, dashboard, rule, event detail hoặc command output.

Detection Logic / Use Cases

  • FIM: thay đổi trong /etc/passwd, /etc/shadow hoặc thư mục cấu hình quan trọng.
  • Privileged command: sudo/su command bất thường ngoài baseline.
  • Authentication: nhiều failed login từ cùng source hoặc cùng username.
  • Suspicious process: command có dấu hiệu download-execute hoặc encoded payload.
# analyst-note
Detection should define: objective → data source → condition → expected evidence → false-positive notes → response action.

Investigation Workflow

  1. Xác định alert type và severity.
  2. Mở raw alert để lấy rule ID, agent name, file path hoặc command line.
  3. Kiểm tra user, host, timestamp, parent process và hành động trước/sau alert.
  4. Đối chiếu với change request hoặc hoạt động lab hợp lệ.
  5. Phân loại: true positive, false positive, policy violation hoặc cần escalate.

Key Findings

  • Wazuh phù hợp để tạo use case SOC cơ bản với chi phí thấp.
  • FIM và auth monitoring tạo ra evidence rõ cho SOC L1.
  • Một số rule cần tuning vì admin activity hợp lệ dễ tạo false positive.

Recommendations

  • Định nghĩa danh sách file critical và user/service account hợp lệ.
  • Tách severity theo asset criticality.
  • Viết playbook riêng cho FIM, auth failure và suspicious command.
  • Thêm enrichment từ CMDB hoặc owner của asset.

Limitations & Next Improvements

  • Viết custom rule cho encoded PowerShell/Bash suspicious pattern.
  • Tích hợp Slack/email notification cho severity cao.
  • Tạo dashboard theo MITRE tactic.

Limitation được ghi rõ để tránh phóng đại kết quả lab. Khi có ảnh/log thật, phần này nên được cập nhật bằng evidence cụ thể.
